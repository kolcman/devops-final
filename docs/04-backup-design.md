# Система резервного копирования

## 1. Назначение и цель

Этот документ описывает систему резервного копирования для инфраструктуры проекта (CA, VPN, Monitoring), а также для инфраструктурных артефактов (скрипты, deb-пакеты, документация).

Цель системы:
- обеспечить восстановление сервисов после сбоев;
- снизить риск потери критичных данных (PKI, конфиги);
- обеспечить воспроизводимое пересоздание инфраструктуры.

## 2. Контур инфраструктуры

Участвующие узлы:
- CA VM (удостоверяющий центр);
- VPN VM (OpenVPN);
- Monitoring VM (Prometheus + Alertmanager);
- Backup Storage A (основное хранилище бэкапов);
- Backup Storage B (резервное хранилище в другом домене отказа).

Принцип хранения:
- минимум 2 копии данных;
- копии размещены в разных доменах отказа (другая VM/зона/облако).

## 3. Политика резервирования

### 3.1 Что резервируем

1) Серверные конфигурации:
- `/etc/openvpn/`
- `/etc/prometheus/`
- `/etc/alertmanager/`
- системные unit-файлы и таймеры, относящиеся к проекту

2) Критичные данные PKI:
- рабочий каталог PKI на CA VM (включая ключи и сертификаты)

3) Артефакты проекта:
- `scripts/`
- `packages/`
- `docs/`
- служебные манифесты релизов (при наличии)

4) Логи бэкапов:
- журнал запусков backup/restore

### 3.2 Что не резервируем

- временные файлы и кэши;
- автоматически переустанавливаемые пакеты ОС;
- секреты, которые по политике должны храниться во внешнем secret-storage (если такой используется).

## 4. Источники и целевые хранилища

Источники:
- CA VM;
- VPN VM;
- Monitoring VM;
- репозиторий проекта.

Цели:
- Backup Storage A: первичное хранилище архивов;
- Backup Storage B: вторичная независимая копия.

Формат данных:
- архивы `tar.gz`;
- контрольные суммы `sha256`;
- метаданные выполнения (timestamp, source host, status).

## 5. Расписание и типы бэкапов

- Daily backup: ежедневно, 02:00 (инкрементальная или файловая копия по текущей реализации скрипта).
- Weekly backup: воскресенье, 03:00 (полный контрольный бэкап).
- On-demand backup: ручной запуск перед критическими изменениями (апдейты OpenVPN/PKI/Prometheus).

Рекомендуемый механизм:
- `systemd timer` как основной вариант;
- `cron` как запасной вариант.

## 6. Retention (политика хранения)

- Daily: хранить 7 последних копий.
- Weekly: хранить 4 последних копии.
- Ежемесячная контрольная копия: хранить 3 копии (опционально, если позволяет место).
- Удаление старых архивов выполняется автоматически скриптом ротации.

## 7. Целевые показатели RPO/RTO

- RPO (допустимая потеря данных): до 24 часов.
- RTO (время восстановления базовой работоспособности): до 4 часов.

Уточнение:
- для сценария полной пересборки инфраструктуры допустим RTO до 1 рабочего дня.

## 8. Автоматизация

Плановые скрипты:
- `scripts/backup/backup-run.sh` — запуск бэкапа;
- `scripts/backup/restore-run.sh` — восстановление;
- `scripts/backup/backup-rotate.sh` — ротация и retention (рекомендуется как отдельный скрипт).

Общие требования к скриптам:
- идемпотентность (повторный запуск не ломает состояние);
- проверка входных параметров и окружения;
- завершение с корректным exit code;
- подробное логирование;
- блокировка параллельного запуска (lock file).

Шаблоны systemd в репозитории:
- `scripts/backup/systemd/linux-final-backup.service`
- `scripts/backup/systemd/linux-final-backup.timer`
- `scripts/backup/systemd/linux-final-backup-rotate.service`
- `scripts/backup/systemd/linux-final-backup-rotate.timer`

Пример установки на VM:

```bash
sudo install -m 0755 scripts/backup/backup-run.sh /usr/sbin/backup-run.sh
sudo install -m 0755 scripts/backup/backup-rotate.sh /usr/sbin/backup-rotate.sh
sudo install -m 0755 scripts/backup/restore-run.sh /usr/sbin/restore-run.sh

sudo install -m 0644 scripts/backup/systemd/linux-final-backup.service /etc/systemd/system/
sudo install -m 0644 scripts/backup/systemd/linux-final-backup.timer /etc/systemd/system/
sudo install -m 0644 scripts/backup/systemd/linux-final-backup-rotate.service /etc/systemd/system/
sudo install -m 0644 scripts/backup/systemd/linux-final-backup-rotate.timer /etc/systemd/system/

sudo systemctl daemon-reload
sudo systemctl enable --now linux-final-backup.timer linux-final-backup-rotate.timer
sudo systemctl list-timers --all | grep linux-final-backup
```

## 9. Безопасность бэкапов

- Доступ к backup-хранилищам только у администраторов.
- PKI-бэкапы хранятся с усиленным контролем доступа.
- Передача между узлами только по защищенному каналу (SSH/SCP/rsync over SSH).
- Контроль целостности каждого архива через `sha256sum`.
- Секреты и пароли не хранятся в репозитории в открытом виде.

## 10. Мониторинг backup-компонента

Минимальные контрольные сигналы:
- время последнего успешного бэкапа;
- статус последнего запуска (success/fail);
- возраст последнего валидного архива;
- наличие свободного места на backup-хранилище.

Источник backup-метрик:
- `scripts/backup/backup-run.sh` пишет метрики в textfile-collector node_exporter:
  - `linux_final_backup_last_run_unixtime`
  - `linux_final_backup_last_run_status`
  - `linux_final_backup_last_success_unixtime`
- путь по умолчанию: `/var/lib/prometheus/node-exporter/linux_final_backup.prom`

Алерты:
- `BackupJobFailed` (critical): последний запуск завершился ошибкой;
- `BackupStale` (critical): нет успешного бэкапа дольше 24 часов;
- `BackupStorageLowSpace` (warning/critical): низкий остаток диска.

Интеграция:
- алерты добавляются в `monitoring/prometheus/alert.rules.yml`;
- уведомления отправляются через существующий Alertmanager.

## 11. Сценарии отказа и план действий

### Сценарий 1: Падение VPN VM

Детект:
- `InstanceDown`/`VPNServerDown` в Prometheus.

Восстановление:
1. Поднять новую VM с базовыми системными настройками.
2. Восстановить конфиги OpenVPN из последнего валидного бэкапа.
3. Восстановить ключевые файлы сервиса.
4. Проверить подключение тестового клиента.

Целевое время: до 2 часов.

### Сценарий 2: Потеря данных на CA VM

Детект:
- ошибки при подписании CSR;
- отсутствие/повреждение PKI-данных.

Восстановление:
1. Развернуть новую CA VM в защищенном сегменте.
2. Восстановить PKI из последней валидной копии.
3. Проверить выпуск тестового сертификата.
4. При компрометации ключа выполнить ротацию сертификатов.

Целевое время: до 4 часов (без массовой ротации клиентов).

### Сценарий 3: Недоступность Backup Storage A

Детект:
- ошибка записи бэкапа;
- недоступность primary storage.

Восстановление:
1. Переключить запись на Backup Storage B.
2. Убедиться в целостности и доступности архивов на B.
3. После восстановления A выполнить обратную синхронизацию.

Целевое время: до 1 часа.

### Сценарий 4: Ошибка backup-скрипта после изменений

Детект:
- неуспешный exit code;
- отсутствие свежего архива;
- алерт `BackupJobFailed`.

Восстановление:
1. Откатить скрипт к последней рабочей версии.
2. Выполнить ручной backup.
3. Исправить причину ошибки и прогнать повторный тест.
4. Задокументировать инцидент и корректирующие действия.

Целевое время: до 1 часа.

### Сценарий 5: Поврежден архив бэкапа

Детект:
- ошибка распаковки;
- несовпадение checksum.

Восстановление:
1. Использовать предыдущую валидную копию.
2. Восстановить сервисы.
3. Выполнить внеплановый полный бэкап.
4. Проверить корректность механизма валидации архивов.

Целевое время: до 2 часов.

### Сценарий 6: Полное пересоздание инфраструктуры

Детект:
- множественный критический отказ;
- текущие VM не подлежат быстрому восстановлению.

Восстановление:
1. Поднять CA, VPN и Monitoring VM.
2. Восстановить конфиги, скрипты и deb-артефакты.
3. Восстановить PKI и мониторинг.
4. Выполнить smoke-тесты:
   - VPN подключение,
   - сбор метрик,
   - доставку алертов.

Целевое время: до 1 рабочего дня.

## 12. Регламент теста восстановления

Периодичность:
- минимум 1 раз в месяц;
- дополнительно после крупных изменений в backup-скриптах.

Чеклист:
1. Взять последнюю валидную копию.
2. Развернуть чистую тестовую VM.
3. Выполнить `restore-run.sh`.
4. Проверить доступность сервисов и целостность данных.
5. Зафиксировать результат в отчете восстановления.

Критерий успешности:
- восстановление выполнено в рамках RTO;
- сервисы работают штатно;
- данные и конфиги консистентны.

## 13. Хранение артефактов проекта

Обязательные артефакты:
- исходники скриптов (`scripts/`);
- исходники deb-пакетов (`packages/`);
- документация (`docs/`);
- инструкции по восстановлению.

Политика:
- основной источник истины — git-репозиторий;
- регулярный бэкап git-репозитория в Backup Storage A/B;
- восстановление возможно без локальной машины разработчика.

## 14. Роли и ответственность

- Администратор инфраструктуры:
  - поддержка backup/restore скриптов;
  - контроль расписания и retention;
  - проведение тестов восстановления.
- Ответственный за мониторинг:
  - поддержка алертов backup-системы;
  - контроль доставки уведомлений.

## 15. Артефакты

Минимальный комплект:
- этот документ `docs/04-backup-design.md`;
- backup/restore/rotation скрипты;
- deb-пакет(ы) для автоматизации backup;
- подтверждение теста восстановления (лог/скрин/краткий отчет);
- подтверждение мониторинга backup-компонента (алерты/метрики).

## 16. Definition of Done

Блок считается завершенным, если:
1. Описана и реализована схема резервирования с двумя независимыми копиями.
2. Настроен автоматический backup по расписанию.
3. Реализованы retention и ротация архивов.
4. Описаны и проверены минимум 5 сценариев отказа (включая полное пересоздание).
5. Настроен мониторинг backup-процесса и алерты.
6. Выполнен минимум один успешный тест восстановления на чистой VM.
